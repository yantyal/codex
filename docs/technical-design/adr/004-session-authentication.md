# ADR-004: Cookie ベースのセッション認証を使う

- 状態: Proposed
- 決定日: 2026-07-11

## コンテキスト

ブラウザ向けの同一オリジン Web アプリであり、MVP では外部 API 利用者やモバイルネイティブアプリを対象にしない。ログアウトやセッション失効を確実に制御したい。

## 決定

不透明で推測困難なセッション ID を `HttpOnly` Cookie で渡し、認証状態とユーザー情報はサーバー側の共有可能な永続ストアへ置く。EKS で API Pod が増えても、特定 Pod へ固定しない。

Cookie は認証状態そのものではなく、セッション ID の配送手段として利用する。`HttpOnly` により Vue の JavaScript から ID を読み取れないようにし、`Secure` により HTTPS 通信だけに限定し、`SameSite` により外部サイトからの意図しない送信を制限する。

## 選定理由

- MVP はブラウザから利用する同一オリジンの Web アプリであり、Cookie と相性がよい。
- サーバー側でセッションを削除すれば、ログアウト、強制失効、パスワード変更時の全端末ログアウトを即時反映できる。
- ブラウザの `localStorage` へトークンを置かないため、XSS が発生した際に JavaScript から認証情報そのものを読み取られる可能性を下げられる。
- JWT の有効期限まで失効できない問題を避けるための denylist や refresh token ローテーションを MVP で管理せずに済む。
- 共有セッションストアを利用すれば、将来 API Pod を複数起動しても sticky session を必要としない。

## 代替案との比較

| 方式                            | 長所                                                         | 今回採用しない理由                                                                       |
| ------------------------------- | ------------------------------------------------------------ | ---------------------------------------------------------------------------------------- |
| JWT を `localStorage` に保存    | API クライアントから明示的に送信しやすい                     | XSS 時にトークンを読み取られやすく、失効と refresh token 管理が複雑になる。              |
| JWT を `HttpOnly` Cookie に保存 | JavaScript から読み取られにくく、分散 API でも自己完結できる | 即時失効、鍵ローテーション、claim 更新の設計が必要で、MVP では自己完結性の利点が小さい。 |
| OAuth/OIDC の外部 IdP           | MFA や組織アカウント連携を任せられる                         | 初期の個人利用には運用と設定が大きい。将来の拡張候補とする。                             |

## 必須対策

- 状態変更 API は CSRF token または同等の対策を行い、`SameSite` だけに依存しない。
- ログイン成功時にセッション ID を再生成し、セッション固定攻撃を防ぐ。
- 有効期限、アイドルタイムアウト、端末ごとの失効を実装する。
- セッション ID の平文を DB やログへ保存せず、漏えい時の影響を抑える。
- 本番では HTTPS と `Secure` Cookie を必須にする。

## 結果

- ブラウザ JavaScript から認証情報へ直接アクセスさせずに済む。
- CSRF 対策、Cookie 属性、セッション固定攻撃対策、共有セッションストアの運用が必要になる。
- JWT の失効管理や refresh token の複雑さを MVP へ持ち込まずに済む。
- 将来、外部 API やネイティブアプリが必要になった場合は別の認可方式を ADR で追加する。
