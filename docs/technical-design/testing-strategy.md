# テスト戦略

## TDD の基本サイクル

1. 利用者に見える振る舞いまたはドメインルールを、失敗するテストとして記述する（Red）。
2. テストを通す最小限の実装を行う（Green）。
3. 重複、命名、責務、依存方向を改善する（Refactor）。
4. 小さな単位でコミットし、テストを常時実行する。

テストのためだけに内部実装を公開せず、公開された振る舞いを検証する。

## テスト構成

| 種類                 | 主な対象                                  | DB・外部依存            | 実行頻度               |
| -------------------- | ----------------------------------------- | ----------------------- | ---------------------- |
| ドメイン単体         | 値オブジェクト、集約、達成率、遅延判定    | なし                    | 保存中、コミット前、CI |
| アプリケーション単体 | ユースケースの流れ、認可、エラー          | port の fake            | 保存中、コミット前、CI |
| インフラ統合         | Prisma Repository、マイグレーション、制約 | Testcontainers の MySQL | コミット前、CI         |
| API 統合             | ルーティング、DTO、認証、HTTP エラー      | テスト用 MySQL          | コミット前、CI         |
| UI コンポーネント    | 入力、バリデーション、状態表示            | API は MSW 等で代替     | 保存中、CI             |
| E2E                  | キャリア作成から評価資料出力まで          | 実コンテナ一式          | PR、リリース前         |

## テスト配置

- 実装に近い単体テストは `*.test.ts` または `*.test.tsx` として同じ機能ディレクトリへ置く。
- MySQL を使う統合テストは `*.integration.test.ts` とし、単体テストと実行コマンドを分ける。
- ブラウザ E2E は `tests/e2e` に置く。
- テストデータ生成は Builder を使い、巨大な共通 fixture への依存を避ける。

## MySQL テスト方針

- SQLite やインメモリ DB で MySQL の代用をしない。
- Repository 統合テストは本番と同じ MySQL メジャーバージョンのコンテナで実行する。
- テスト開始時にマイグレーションを適用する。
- 各テストはトランザクション rollback またはスキーマ初期化で独立させる。
- 一意制約、外部キー、日時精度、照合順序を統合テストで確認する。

## 最初に書く重要テスト

1. 別ユーザーのキャリア目標を取得・更新できない。
2. 数値、マイルストーン、習慣、手動の達成率が業務ルールどおりになる。
3. 達成率 100% と超過率を分けて表示できる。
4. ロードマップに循環依存を登録できない。
5. 日次実績の未来日と 1,440 分超過を拒否する。
6. 評価資料が指定期間と所有ユーザーのデータだけを集約する。
7. 元データ更新後も確定した評価資料のスナップショットが変わらない。

## カバレッジ

- 数値目標を目的化せず、重要な分岐と業務ルールの網羅を優先する。
- ドメイン層は原則として分岐をすべてテストする。
- 新規・変更コードはテストなしでマージしない。例外は設定・文書のみとし、理由を PR に記載する。
- CI では format、lint、型検査、単体、統合、E2E、依存関係監査の順に実行する。

## チケットの完了条件

- 受入条件が自動テストまたは明示的な確認手順に対応している。
- 正常系だけでなく、入力境界、未認証、他ユーザー所有、競合を確認している。
- テスト名が「どの条件で何が起きるか」を説明している。
- テストが実行順序、ローカル時刻、共有 DB 状態に依存しない。
