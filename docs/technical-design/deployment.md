# ローカル実行・EKS 移行

## ローカル構成

Docker Compose で次のサービスを起動する。

| サービス | 役割                             | 公開ポートの例                            |
| -------- | -------------------------------- | ----------------------------------------- |
| `web`    | Vue 3 開発サーバーまたは静的配信 | 3000                                      |
| `api`    | Express REST API                 | 3001（ブラウザからは web 経由）           |
| `mysql`  | MySQL DB                         | 原則としてホストへ非公開。開発時のみ 3306 |

DB データは named volume へ保存する。ソースコードは開発時のみ bind mount し、本番相当の確認ではビルド済みイメージを利用する。

## 環境変数

| 変数             | 用途                       | 秘密情報         |
| ---------------- | -------------------------- | ---------------- |
| `NODE_ENV`       | 実行環境                   | いいえ           |
| `PORT`           | API 待受ポート             | いいえ           |
| `DATABASE_URL`   | MySQL 接続文字列           | はい             |
| `SESSION_SECRET` | セッション署名・暗号用     | はい             |
| `APP_ORIGIN`     | 許可する Web オリジン      | いいえ           |
| `LOG_LEVEL`      | ログレベル                 | いいえ           |
| `TZ`             | プロセスの基準タイムゾーン | いいえ。原則 UTC |

`.env.example` にはダミー値だけを置く。実値の `.env` は Git 管理しない。設定は起動時に検証し、不足している場合は明確なエラーで終了する。

## マイグレーション

- Prisma migration をソース管理する。
- 開発環境では明示的なコマンドでマイグレーションを作成・適用する。
- 本番相当環境では API の各 Pod が自動実行せず、デプロイ前の単一 Job として適用する。
- 破壊的変更は expand-and-contract を基本とし、ロールバック手順とバックアップ確認を用意する。
- seed は開発・テスト用と本番用を分離する。

## ヘルスチェック

- `/health/live`: プロセスが応答できるかだけを確認する。DB 障害で失敗させない。
- `/health/ready`: MySQL 接続と必須依存を確認する。
- 詳細なエラー、接続文字列、秘密情報はレスポンスへ含めない。

## ログ・監視

- JSON 構造化ログを標準出力へ出す。
- `traceId`、HTTP メソッド、パス、ステータス、処理時間を記録する。
- パスワード、Cookie、本文、評価内容、個人情報を記録しない。
- 将来 EKS でメトリクスとトレースを収集できるよう、ロガーと計測処理を差し替え可能にする。

## EKS 移行時の考慮

現在の `k8s/`、`infra/terraform/aws-eks/`、Docker 関連資産は、この設計段階では変更しない。実装時に次を段階的に更新する。

- Web と API の Deployment／Service を分離する。
- SQLite PVC を廃止し、Amazon RDS for MySQL など外部 DB へ接続する。
- `DATABASE_URL` と `SESSION_SECRET` を Kubernetes Secret または外部 Secret 管理へ移す。
- readiness／liveness probe を新しいヘルスチェックへ向ける。
- API を複数 replica で動かせるよう、ローカルファイルとプロセスメモリへ状態を置かない。
- PDF や将来の添付ファイルは Pod ローカルへ永続化せず、再生成またはオブジェクトストレージを利用する。
- マイグレーションをデプロイ前 Job として一度だけ実行する。

## バックアップ

- ローカルでは MySQL dump と volume の扱いを手順化する。
- EKS 移行後は RDS の自動バックアップと Point-in-Time Recovery を利用する構成を検討する。
- 復元できることを定期的に別環境で確認する。バックアップ作成だけで完了としない。
