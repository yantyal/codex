# Codex 作業ルール

このファイルは、このプロジェクトで Codex が作業するときの基本ルールをまとめたものです。

## 作業開始前の最新化確認

新しい作業を始める前に、ローカルの状態とリモートとの差分を確認する。
これにより、古いブランチや古いファイルを元に作業してデグレを起こすことを防ぐ。

1. 現在のブランチと作業ツリーの状態を確認する。

```powershell
git status --short --branch
```

2. リモートの最新情報を取得する。

```powershell
git fetch --prune
```

3. 現在のブランチがリモートより遅れていないか確認する。

```powershell
git status --short --branch
```

4. `behind` と表示された場合は、作業前に最新化する。

```powershell
git pull --ff-only
```

5. `ahead`、`behind`、`diverged` の表示がある場合は、作業前に状況を説明する。
   特に `diverged` の場合は、勝手にマージや rebase をせず、方針を確認する。

6. 新しい作業でブランチを作る場合は、最新化した既定ブランチから作成する。

```powershell
git switch main
git pull --ff-only
git switch -c feature/example
```

ネットワーク制限などで `git fetch` や `git pull` が失敗した場合は、そのまま進めず、最新化できていないことを説明する。

## リリースノートとタグのルール

変更を行う場合は、原則としてリリースノートの更新、バージョン更新、Git タグ作成までをセットで検討する。
これにより、どの変更がどのバージョンに入ったかをあとから追いやすくする。

- 変更内容を `CHANGELOG.md` に追記する。
- 変更の大きさに応じて `package.json` と `package-lock.json` の `version` を更新する。
- バージョン番号は SemVer に従い、`MAJOR.MINOR.PATCH` で管理する。
- 変更をコミットしたあと、対応する注釈付き Git タグを作成する。
- タグ名は `v1.1.0` のように、先頭に `v` を付ける。
- タグ作成や push の詳しい手順は `docs/versioning.md` を確認する。
- リリース対象ではない変更として扱う場合は、リリースノートやタグを更新しない理由を作業結果に明記する。

例:

```powershell
npm version patch --no-git-tag-version
git add package.json package-lock.json CHANGELOG.md
git commit -m "Release v1.1.1"
git tag -a v1.1.1 -m "Release v1.1.1"
```

## 基本方針

- 既存コードや設定を大きく壊さないように、必要最小限の変更を行う。
- 変更する前に、関連するファイルを確認してから作業する。
- 変更後は、可能であればテストや動作確認を行う。
- 実行したコマンドや変更した理由は、簡潔に説明する。
- README やコメントは、初心者にも分かる言葉で書く。

## コメント記述ルール

コメントは、あとから読む人がコードの役割や意図を理解しやすくするために書く。
処理内容だけでなく、必要に応じて「なぜその処理をするのか」も説明する。

- コメントは初心者にも分かりやすい内容にする。
- 専門用語を使う場合は、初心者にも分かる表現にする。
- コメントの文末は「します」ではなく「する」に統一する。
- クラスコメントは必ず書き、そのクラスの役割を書く。
- メソッドコメントは必ず書き、そのメソッドの目的、引数、戻り値、注意点を書く。

## ファイル命名ルール

ファイル名や拡張子をそろえると、探す場所や書き方で迷いにくくなる。
特に YAML ファイルは拡張子が複数あるため、このプロジェクトでは `.yaml` に統一する。

- YAML ファイルの拡張子は `.yml` ではなく `.yaml` に統一する。
- 新規作成する YAML ファイルは必ず `.yaml` にする。
- 既存の `.yml` ファイルがある場合は、参照箇所も確認して `.yaml` にリネームする。

## Windows 環境での注意

- このプロジェクトは Windows 環境で作業される前提を考慮する。
- コマンド例を書くときは、PowerShell で実行しやすい形を優先する。
- パス区切りや環境変数の書き方が OS によって違う場合は、Windows で分かりやすい説明を加える。

## 不明点がある場合

- 仕様や目的が不明な場合は、勝手に大きな仕様変更をしない。
- 判断に迷う変更は、作業前に確認する。
- 既存の重要な説明や設定は、理由なく削除しない。
