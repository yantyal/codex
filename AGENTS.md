# Codex 作業ルール

このファイルは、このプロジェクトで Codex が作業するときの基本ルールをまとめたものです。

## 基本方針

- 既存コードや設定を大きく壊さないように、必要最小限の変更を行います。
- 変更する前に、関連するファイルを確認してから作業します。
- 変更後は、可能であればテストや動作確認を行います。
- 実行したコマンドや変更した理由は、簡潔に説明します。
- README やコメントは、初心者にも分かる言葉で書きます。

## コメント記述ルール

コメントは、あとから読む人がコードの役割や意図を理解しやすくするために書きます。
処理内容だけでなく、必要に応じて「なぜその処理をするのか」も説明します。

- コメントは初心者にも分かりやすい内容にする。
- 専門用語を使う場合は、初心者にも分かる表現にする。
- コメントの文末は「します」ではなく「する」に統一する。
- クラスコメントは必ず書き、そのクラスの役割を書く。
- メソッドコメントは必ず書き、そのメソッドの目的、引数、戻り値、注意点を書く。

## ファイル命名ルール

ファイル名や拡張子をそろえると、探す場所や書き方で迷いにくくなります。
特に YAML ファイルは拡張子が複数あるため、このプロジェクトでは `.yaml` に統一します。

- YAML ファイルの拡張子は `.yml` ではなく `.yaml` に統一します。
- 新規作成する YAML ファイルは必ず `.yaml` にします。
- 既存の `.yml` ファイルがある場合は、参照箇所も確認して `.yaml` にリネームします。

## Windows 環境での注意

- このプロジェクトは Windows 環境で作業される前提を考慮します。
- コマンド例を書くときは、PowerShell で実行しやすい形を優先します。
- パス区切りや環境変数の書き方が OS によって違う場合は、Windows で分かりやすい説明を加えます。

## 不明点がある場合

- 仕様や目的が不明な場合は、勝手に大きな仕様変更をしません。
- 判断に迷う変更は、作業前に確認します。
- 既存の重要な説明や設定は、理由なく削除しません。
